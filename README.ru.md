# MyPersonalGit

[![.NET 10](https://img.shields.io/badge/.NET-10.0-512BD4?logo=dotnet&logoColor=white)](https://dotnet.microsoft.com/) [![Blazor Server](https://img.shields.io/badge/Blazor-Server-512BD4?logo=blazor&logoColor=white)](https://dotnet.microsoft.com/apps/aspnet/web-apps/blazor) [![SQLite](https://img.shields.io/badge/SQLite-Default-003B57?logo=sqlite&logoColor=white)](https://www.sqlite.org/) [![PostgreSQL](https://img.shields.io/badge/PostgreSQL-Optional-4169E1?logo=postgresql&logoColor=white)](https://www.postgresql.org/) [![Docker](https://img.shields.io/badge/Docker-Hub-2496ED?logo=docker&logoColor=white)](https://hub.docker.com/r/fennch/mypersonalgit) [![CI/CD](https://img.shields.io/badge/CI%2FCD-Auto_Release-brightgreen?logo=githubactions&logoColor=white)](#ci-cd) [![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE) [![GitHub last commit](https://img.shields.io/github/last-commit/ChrisDFennell/MyPersonalGit)](https://github.com/ChrisDFennell/MyPersonalGit)

Самостоятельно размещаемый Git-сервер с веб-интерфейсом в стиле GitHub, построенный на ASP.NET Core и Blazor Server. Просматривайте репозитории, управляйте задачами, pull-запросами, вики, проектами и многим другим — всё на вашем собственном компьютере или сервере.

## Возможности

### Код и репозитории
- **Управление репозиториями** — Создание, просмотр и удаление Git-репозиториев с полным обозревателем кода, редактором файлов, историей коммитов, ветками и тегами
- **Импорт/миграция репозиториев** — Импорт репозиториев из GitHub, GitLab, Bitbucket, Gitea/Forgejo/Gogs или по любому Git URL с возможностью импорта задач и PR. Фоновая обработка с отслеживанием прогресса
- **Архивирование репозиториев** — Пометка репозиториев как «только для чтения» с визуальными значками; push-операции блокируются для архивных репозиториев
- **Git Smart HTTP** — Клонирование, получение и отправка по HTTP с Basic Auth
- **Встроенный SSH-сервер** — Собственный SSH-сервер для Git-операций — внешний OpenSSH не требуется. Поддержка обмена ключами ECDH, шифрования AES-CTR и аутентификации по открытому ключу (RSA, ECDSA, Ed25519)
- **Аутентификация по SSH-ключам** — Добавляйте SSH-ключи к вашей учётной записи и выполняйте Git-операции через SSH с автоматическим управлением `authorized_keys` (или через встроенный SSH-сервер)
- **Форки и синхронизация с upstream** — Создание форков, синхронизация с upstream одним щелчком и отображение связей форков в интерфейсе
- **Git LFS** — Поддержка Large File Storage для отслеживания бинарных файлов
- **Зеркалирование репозиториев** — Зеркалирование репозиториев на/из внешних Git-удалённых серверов
- **Сравнение** — Сравнение веток с подсчётом коммитов «впереди/позади» и полным отображением различий
- **Статистика по языкам** — Полоска разбивки по языкам в стиле GitHub на каждой странице репозитория
- **Защита веток** — Настраиваемые правила для обязательных ревью, проверок статуса, предотвращения force-push и подтверждения от CODEOWNERS
- **Обязательная подпись коммитов** — Правило защиты ветки, требующее GPG-подписи всех коммитов перед слиянием
- **Защита тегов** — Защита тегов от удаления, принудительного обновления и несанкционированного создания с помощью glob-шаблонов и списков разрешённых пользователей
- **Проверка подписей коммитов** — Верификация GPG-подписей на коммитах и аннотированных тегах с отметками «Verified» / «Signed» в интерфейсе
- **Метки репозитория** — Управление метками с настраиваемыми цветами для каждого репозитория; метки автоматически копируются при создании репозиториев из шаблонов
- **AGit Flow** — Рабочий процесс push-to-review: `git push origin HEAD:refs/for/main` создаёт pull-запрос без форка или создания удалённых веток. Обновляет существующие открытые PR при последующих push-операциях
- **Обзор** — Просмотр всех доступных репозиториев с поиском, сортировкой и фильтрацией по темам
- **Star from Explore** — Добавляйте и убирайте звёзды у репозиториев прямо со страницы Обзор, не открывая каждый репозиторий
- **Autolink References** — Автоматическое преобразование `#123` в ссылки на задачи, а также настраиваемые пользовательские шаблоны (например, `JIRA-456` → внешние URL) для каждого репозитория
- **Поиск** — Полнотекстовый поиск по репозиториям, задачам, PR и коду
- **License Detection** — Автоматическое обнаружение файлов LICENSE и идентификация распространённых лицензий (MIT, Apache-2.0, GPL, BSD, ISC, MPL, Unlicense) с бейджем в боковой панели репозитория

### Совместная работа
- **Задачи и Pull-запросы** — Создание, комментирование, закрытие/переоткрытие задач и PR с метками, множественными назначенными, сроками и ревью. Слияние PR через merge commit, squash или rebase. Веб-разрешение конфликтов слияния с двухпанельным просмотром различий
- **Зависимости задач** — Определение отношений «заблокировано» и «блокирует» между задачами с обнаружением циклических зависимостей
- **Закрепление и блокировка задач** — Закрепление важных задач вверху списка и блокировка обсуждений для предотвращения дальнейших комментариев
- **Редактирование и удаление комментариев** — Редактирование или удаление собственных комментариев к задачам и pull-запросам с индикатором «(изменено)»
- **@Mention Notifications** — @упоминайте пользователей в комментариях для отправки им прямого уведомления
- **Разрешение конфликтов слияния** — Разрешение конфликтов слияния прямо в браузере с визуальным редактором, показывающим base/ours/theirs, кнопками быстрого принятия и валидацией маркеров конфликтов
- **Squash Commit Message** — Настройте сообщение коммита при squash-слиянии pull request
- **Branch Delete After Merge** — Опция автоматического удаления исходной ветки после слияния pull request, включена по умолчанию
- **Обсуждения** — Ветвящиеся обсуждения в стиле GitHub Discussions для каждого репозитория с категориями (Общее, Вопросы и ответы, Объявления, Идеи, Показать и рассказать, Голосования), закреплением/блокировкой, пометкой как ответ и голосованием
- **Предложения при ревью кода** — Режим «Предложить изменения» в инлайн-ревью PR позволяет рецензентам предлагать замены кода прямо в diff
- **Image Diff** — Сравнение изображений бок о бок в pull-запросах с ползунком непрозрачности для визуального сравнения изменённых изображений (PNG, JPG, GIF, SVG, WebP)
- **File Tree в PR** — Сворачиваемая боковая панель с деревом файлов в представлении diff pull-запроса для удобной навигации между изменёнными файлами
- **Отметка файлов как просмотренных** — Отслеживание прогресса ревью в pull-запросах с чекбоксами «Просмотрено» для каждого файла и счётчиком прогресса
- **Подсветка синтаксиса в diff** — Языковая подсветка синтаксиса в diff pull-запросов и сравнений через Prism.js
- **Реакции эмодзи** — Реакции на задачи, PR, обсуждения и комментарии: одобрение, неодобрение, сердце, смех, ура, замешательство, ракета и глаза
- **Auto-Merge** — Включите автоматическое слияние в pull-запросах для автоматического слияния, когда все обязательные проверки статуса пройдены и ревью одобрены
- **CI Status on PR List** — Список pull-запросов показывает зелёные/красные/жёлтые иконки статуса CI рядом с каждым заголовком PR
- **Cherry-Pick / Revert via UI** — Перенесите любой коммит в другую ветку (cherry-pick) или отмените коммит напрямую или как новый pull-запрос через веб-интерфейс
- **Transfer Issues** — Перемещайте задачи между репозиториями с сохранением заголовка, описания, комментариев, совпадающих меток и ссылкой на оригинал с заметкой о переносе
- **Saved Replies** — Сохраняйте шаблонные ответы и быстро вставляйте их при комментировании задач или pull request
- **Batch Issue Operations** — Выбирайте несколько задач и закрывайте или переоткрывайте их массово из списка задач
- **CODEOWNERS** — Автоназначение рецензентов PR по путям файлов с возможностью требовать подтверждение от CODEOWNERS перед слиянием
- **Шаблоны репозиториев** — Создание новых репозиториев из шаблонов с автоматическим копированием файлов, меток, шаблонов задач и правил защиты веток
- **Черновики задач и шаблоны задач** — Создание черновых задач (в процессе) и определение многоразовых шаблонов задач (отчёт об ошибке, запрос функции) для каждого репозитория с метками по умолчанию
- **PR Template** — Автоматическое заполнение описаний pull request из `.github/PULL_REQUEST_TEMPLATE.md`
- **Release Editing** — Редактирование заголовков, описаний и флагов черновика/предрелиза у релизов после создания
- **Вики** — Вики-страницы на Markdown для каждого репозитория с историей ревизий
- **Проекты** — Kanban-доски с перетаскиванием карточек для организации работы
- **Сниппеты** — Обмен фрагментами кода (как GitHub Gists) с подсветкой синтаксиса и множественными файлами
- **Организации и команды** — Создание организаций с участниками и командами, назначение командных прав доступа к репозиториям
- **Гранулярные права доступа** — Пятиуровневая модель прав (Read, Triage, Write, Maintain, Admin) для тонкой настройки контроля доступа к репозиториям
- **Вехи** — Отслеживание прогресса задач по вехам с индикаторами прогресса и сроками
- **Комментарии к коммитам** — Комментарии к отдельным коммитам с опциональной привязкой к файлу/строке
- **Темы репозитория** — Присвоение тем репозиториям для обнаружения и фильтрации на странице «Обзор»
- **Activity Pulse** — Еженедельная сводная страница по репозиторию, показывающая объединённые PR, открытые/закрытые задачи, коммиты, топ-контрибьюторов и активные ветки за последние 7 дней

### Web IDE
- **Полнофункциональный редактор кода** — IDE на базе Monaco Editor с подсветкой синтаксиса для более 30 языков, несколькими темами, раскрытием сокращений Emmet для HTML/CSS и форматированием при вставке
- **Language Server Protocol (LSP)** — IntelliSense, автодополнение, информация при наведении, переход к определению, поиск ссылок, справка по сигнатурам, форматирование кода, переименование символа, действия с кодом и диагностика в реальном времени для 12 языков: C# (OmniSharp), TypeScript/JavaScript, Python, Go, Rust, HTML, CSS, JSON, YAML, Bash, Dockerfile и Markdown
- **AI-дополнение кода** — Встроенные подсказки-призраки при вводе, работающие через любой OpenAI-совместимый API (OpenAI, Ollama, LM Studio и др.). Настраиваемые эндпоинт, API-ключ и модель в настройках администратора. Tab для принятия
- **AI-чат и ассистент** — Встроенная панель AI-чата с контекстно-зависимыми разговорами. Щёлкните правой кнопкой по коду для AI-действий: Объяснить код, Рефакторинг кода, Сгенерировать тесты, Исправить код, Добавить документацию и Встроенное объяснение (Ctrl+Shift+E показывает виджет объяснения у курсора). Потоковые ответы с действиями для блоков кода: применить/скопировать/создать файл
- **AI-генерация тестов** — Выберите код и сгенерируйте комплексные модульные тесты с автоопределением фреймворка (xUnit, Jest, pytest, Go testing, Rust #[cfg(test)], JUnit 5, RSpec, PHPUnit)
- **AI-рефакторинг** — Выберите код и получите предложения по рефакторингу на основе AI с применением в один клик для замены выделенного фрагмента
- **Встроенный отладчик** — Мультиязычная отладка через DAP (Debug Adapter Protocol) с точками останова (клик по полю), шаг через/в/из, продолжить, инспекция переменных, навигация по стеку вызовов и вывод консоли отладки. Поддержка Python (debugpy), C# (netcoredbg), JavaScript/TypeScript (js-debug), Go (Delve), Rust (lldb-dap) и Java (JDWP). Автоматическое определение языка по расширению файла
- **Панель проблем** — Вкладка нижней панели, агрегирующая все диагностики LSP (ошибки/предупреждения) открытых файлов, сгруппированные по файлам с переходом по клику. Счётчики ошибок/предупреждений в строке состояния
- **Запуск задач** — Вкладка нижней панели для запуска команд сборки/тестирования (dotnet build, npm test, go build, make, pytest и др.) с потоковым выводом в реальном времени, кликабельными расположениями ошибок и поддержкой отмены
- **Сравнение веток** — Панель боковой панели для сравнения двух веток бок о бок. Показывает список изменённых файлов со статусом добавления/изменения/удаления/переименования, клик для просмотра различий в diff-редакторе Monaco
- **Встроенные маркеры различий** — Зелёные, синие и красные индикаторы на полях редактора, показывающие добавленные, изменённые и удалённые строки по сравнению с последним коммитом. Нажмите на полоску полей, чтобы увидеть оригинальное содержимое встроенно
- **Code Snippets** — Пользовательские фрагменты кода, хранящиеся в браузере. Определите префикс, язык и тело в Settings; введите префикс и нажмите Tab для раскрытия. Поддерживается синтаксис сниппетов VS Code ($1, $2, $0 плейсхолдеры)
- **Управление файлами** — Иерархическое дерево файлов с цветными иконками по языку (более 40 типов файлов), вложенностью файлов (группирует `.razor` + `.razor.css` + `.razor.cs`), поиск/фильтрация, загрузка файлов перетаскиванием и контекстные меню для создания файла/папки/переименования/удаления
- **Управление вкладками** — Многовкладочный интерфейс с цветными иконками файлов, перетаскиванием для перестановки, закреплёнными вкладками, подтверждением закрытия при несохранённых изменениях, отмена закрытия вкладки (Ctrl+Shift+T), контекстным меню правой кнопки (Закрыть, Закрыть другие, Закрыть справа, Закрыть сохранённые) и стрелками прокрутки при переполнении
- **Разделённый редактор и просмотр различий** — Параллельное редактирование с независимой прокруткой и просмотр различий для сравнения изменений перед коммитом
- **Встроенный терминал** — Терминал на базе xterm.js с несколькими вкладками терминала, доступом к оболочке через WebSocket и светлой/тёмной темой в зависимости от настроек
- **Интеграция с Git** — Создание веток, просмотр blame, история файлов, панель коммитов, управление исходным кодом с выбором файлов и визуальный граф коммитов с цветными линиями дорожек и метками веток
- **Разрешение конфликтов слияния** — Встроенные кнопки «Принять текущий» / «Принять входящий» / «Принять оба» с цветовыми зонами конфликтов (зелёный/синий)
- **Поиск и замена** — Глобальный поиск по всем файлам с фильтрацией по расширению, построчными результатами и заменой всех вхождений
- **Навигация по коду** — Быстрое открытие (Ctrl+P), Палитра команд (Ctrl+Shift+P), Перейти к строке (Ctrl+G), Go to Definition (F12), Peek Definition (Alt+F12), Find References (Shift+F12), Rename Symbol (F2), панель структуры/символов и навигация по хлебным крошкам

### CI/CD и DevOps
- **CI/CD Runner** — Определение рабочих процессов в `.github/workflows/*.yml` и запуск в Docker-контейнерах. Автоматический запуск при push и pull request
- **Совместимость с GitHub Actions** — Один и тот же YAML-файл работает как в MyPersonalGit, так и в GitHub Actions. Транслирует действия `uses:` (`actions/checkout`, `actions/setup-dotnet`, `actions/setup-node`, `actions/setup-python`, `actions/setup-java`, `docker/login-action`, `docker/build-push-action`, `softprops/action-gh-release`) в эквивалентные shell-команды
- **Параллельные задачи с `needs:`** — Задачи объявляют зависимости через `needs:` и выполняются параллельно при отсутствии зависимостей. Зависимые задачи ждут завершения предшествующих и автоматически отменяются при сбое зависимости
- **Условные шаги (`if:`)** — Шаги поддерживают выражения `if:`: `always()`, `success()`, `failure()`, `cancelled()`, `true`, `false`. Шаги очистки с `if: failure()` или `if: always()` выполняются даже после предыдущих сбоев
- **Выходные данные шагов (`$GITHUB_OUTPUT`)** — Шаги могут записывать пары `key=value` или многострочные `key<<DELIMITER` в `$GITHUB_OUTPUT`, а последующие шаги получают их как переменные окружения, совместимо с синтаксисом `${{ steps.X.outputs.Y }}`
- **Контекст `github`** — `GITHUB_SHA`, `GITHUB_REF`, `GITHUB_REF_NAME`, `GITHUB_ACTOR`, `GITHUB_REPOSITORY`, `GITHUB_EVENT_NAME`, `GITHUB_WORKSPACE`, `GITHUB_RUN_ID`, `GITHUB_JOB`, `GITHUB_WORKFLOW` и `CI=true` автоматически внедряются в каждую задачу
- **Матричные сборки** — `strategy.matrix` расширяет задачи по нескольким комбинациям переменных (например, ОС x версия). Поддерживает `fail-fast` и подстановку `${{ matrix.X }}` в `runs-on`, командах шагов и именах шагов
- **Входные параметры `workflow_dispatch`** — Ручной запуск с типизированными входными параметрами (string, boolean, choice, number). UI показывает форму ввода при запуске рабочих процессов с параметрами. Значения внедряются как переменные окружения `INPUT_*`
- **Тайм-ауты задач (`timeout-minutes`)** — Установка `timeout-minutes` для задач для автоматического сбоя при превышении лимита. По умолчанию: 360 минут (как в GitHub Actions)
- **`if:` на уровне задач** — Пропуск целых задач по условиям. Задачи с `if: always()` выполняются даже при сбое зависимостей. Пропущенные задачи не вызывают сбой запуска
- **Выходные данные задач** — Задачи объявляют `outputs:`, которые зависимые задачи через `needs:` используют через `${{ needs.X.outputs.Y }}`. Выходные данные разрешаются из выходных данных шагов после завершения задачи
- **`continue-on-error`** — Отметка отдельных шагов как допускающих сбой без провала всей задачи. Полезно для необязательной валидации или уведомлений
- **Фильтр `on.push.paths`** — Запуск рабочих процессов только при изменении определённых файлов. Поддержка glob-шаблонов (`src/**`, `*.ts`) и `paths-ignore:` для исключений
- **Перезапуск рабочих процессов** — Перезапуск неудачных, успешных или отменённых запусков одним щелчком из интерфейса Actions. Создаёт свежий запуск с той же конфигурацией
- **`working-directory`** — Установка `defaults.run.working-directory` на уровне рабочего процесса или `working-directory:` для каждого шага для управления директорией выполнения команд
- **`defaults.run.shell`** — Настройка shell для рабочего процесса или шага (`bash`, `sh`, `python3` и т.д.)
- **`strategy.max-parallel`** — Ограничение количества одновременно выполняемых матричных задач
- **Reusable Workflows (`workflow_call`)** — Определяйте рабочие процессы с `on: workflow_call`, которые другие процессы могут вызывать через `uses: ./.github/workflows/build.yml`. Поддерживает типизированные входные данные, выходные данные и секреты. Задачи вызванного процесса встраиваются в вызывающий
- **Composite Actions** — Определяйте многошаговые действия в `.github/actions/{name}/action.yml` с `runs: using: composite`. Шаги составных действий раскрываются встроенно при выполнении
- **Environment Deployments** — Настраивайте среды развёртывания (напр., `staging`, `production`) с правилами защиты: обязательные рецензенты, таймеры ожидания и ограничения веток. Задачи рабочего процесса с `environment:` требуют одобрения перед выполнением. Полная история развёртываний с интерфейсом одобрения/отклонения
- **`on.workflow_run`** — Цепочки рабочих процессов: запуск процесса B при завершении процесса A. Фильтрация по имени рабочего процесса и `types: [completed]`
- **Автоматическое создание релизов** — `softprops/action-gh-release` создаёт реальные сущности Release с тегом, заголовком, телом changelog и флагами pre-release/draft. Архивы исходного кода (ZIP и TAR.GZ) автоматически прикрепляются как загружаемые ассеты
- **Конвейер автоматического релиза** — Встроенный рабочий процесс автоматически тегирует версии, генерирует changelog и отправляет Docker-образы в Docker Hub при каждом push в main
- **Проверки статуса коммитов** — Рабочие процессы автоматически устанавливают статус pending/success/failure для коммитов, видимый в pull-запросах
- **Отмена рабочих процессов** — Отмена выполняющихся или ожидающих рабочих процессов из интерфейса Actions
- **Контроль параллелизма** — Новые push-операции автоматически отменяют ожидающие запуски того же рабочего процесса
- **Переменные окружения рабочих процессов** — Установка `env:` на уровне рабочего процесса, задачи или шага в YAML
- **Статусные значки** — Встраиваемые SVG-значки для статуса рабочего процесса и коммита (`/api/badge/{repo}/workflow`)
- **Загрузка артефактов** — Загрузка артефактов сборки прямо из интерфейса Actions
- **Управление секретами** — Зашифрованные секреты репозитория (AES-256), внедряемые как переменные окружения в запуски CI/CD
- **Вебхуки** — Запуск внешних сервисов при событиях в репозитории
- **Метрики Prometheus** — Встроенная конечная точка `/metrics` для мониторинга

### Аутентификация
- **OAuth2 / SSO** — Вход через GitHub, Google, Microsoft, GitLab, Bitbucket, Facebook, Discord или Twitter/X. Администраторы настраивают Client ID и Secret для каждого провайдера в панели администратора — пользователям отображаются только провайдеры с заполненными учётными данными
- **Провайдер OAuth2** — Работа в качестве провайдера идентификации, чтобы другие приложения могли использовать «Войти через MyPersonalGit». Реализация Authorization Code flow с PKCE, обновление токенов, конечная точка userinfo и OpenID Connect discovery (`.well-known/openid-configuration`)
- **LDAP / Active Directory** — Аутентификация пользователей через LDAP-каталог или домен Active Directory. Пользователи автоматически создаются при первом входе с синхронизацией атрибутов (email, отображаемое имя). Поддержка повышения до администратора по группе, SSL/TLS и StartTLS
- **SSPI / Встроенная аутентификация Windows** — Прозрачный Single Sign-On для доменных пользователей Windows через Negotiate/NTLM. Пользователи в домене аутентифицируются автоматически без ввода учётных данных. Включается в Admin > Settings (только Windows)
- **Двухфакторная аутентификация** — TOTP-аутентификация через приложение-аутентификатор с кодами восстановления
- **WebAuthn / Passkeys** — Поддержка аппаратных ключей безопасности FIDO2 и passkeys в качестве второго фактора. Регистрация YubiKey, платформенных аутентификаторов (Face ID, Windows Hello, Touch ID) и других FIDO2-устройств. Проверка счётчика подписей для обнаружения клонированных ключей
- **Привязанные аккаунты** — Пользователи могут привязать несколько OAuth-провайдеров к своей учётной записи в настройках

- **User-Level Secrets** — Зашифрованные секреты, общие для всех репозиториев пользователя, управляемые в Settings > Secrets
- **Organization-Level Secrets** — Зашифрованные секреты, общие для всех репозиториев организации, управляемые на вкладке Secrets организации
- **Repository Pinning** — Закрепление до 6 избранных репозиториев на странице профиля пользователя для быстрого доступа
- **Git Hooks Management** — Веб-интерфейс для просмотра, редактирования и управления серверными Git Hooks (pre-receive, update, post-receive, post-update, pre-push) для каждого репозитория
- **Protected File Patterns** — Правило защиты ветки с glob-шаблонами для требования одобрения рецензента при изменении определённых файлов (например, `*.lock`, `migrations/**`, `.github/workflows/*`)
- **External Issue Tracker** — Настройка репозиториев для перенаправления на внешний трекер задач (Jira, Linear и др.) с пользовательскими шаблонами URL
- **Federation (NodeInfo/WebFinger)** — Обнаружение NodeInfo 2.0, WebFinger и host-meta для межинстансной обнаруживаемости
- **Distributed CI Runners** — Внешние раннеры могут регистрироваться через API, опрашивать очередь заданий и сообщать результаты

### 10. Аутентификация по SSH-ключам

Используйте SSH-ключи для Git-операций без пароля. Есть два варианта:

#### Вариант A: Встроенный SSH-сервер (рекомендуется)

Внешний SSH-демон не требуется — MyPersonalGit запускает собственный SSH-сервер:

1. Перейдите в **Admin > Settings** и включите **Built-in SSH Server**
2. Установите SSH-порт (по умолчанию: 2222) — используйте 22, если системный SSH не запущен
3. Сохраните настройки и перезапустите сервер (изменение порта требует перезапуска)
4. Перейдите в **Settings > SSH Keys** и добавьте ваш публичный ключ (`~/.ssh/id_ed25519.pub`, `~/.ssh/id_rsa.pub` или `~/.ssh/id_ecdsa.pub`)
5. Клонирование через SSH:
   ```bash
   git clone ssh://youruser@yourserver:2222/MyRepo.git
   ```

Встроенный SSH-сервер поддерживает обмен ключами ECDH-SHA2-NISTP256, шифрование AES-128/256-CTR, HMAC-SHA2-256 и аутентификацию по открытому ключу с ключами Ed25519, RSA и ECDSA.

#### Вариант B: Системный OpenSSH

Если вы предпочитаете использовать системный SSH-демон:

1. Перейдите в **Settings > SSH Keys** и добавьте ваш публичный ключ
2. MyPersonalGit автоматически поддерживает файл `authorized_keys` из всех зарегистрированных SSH-ключей
3. Настройте OpenSSH вашего сервера на использование сгенерированного файла authorized_keys:
   ```
   # В /etc/ssh/sshd_config
   AuthorizedKeysFile /path/to/.mypersonalgit/ssh/authorized_keys
   ```
4. Клонирование через SSH:
   ```bash
   git clone ssh://git@yourserver:22/repos/MyRepo.git
   ```

Сервис SSH-аутентификации также предоставляет API по адресу `/api/ssh/authorized-keys` для использования с директивой `AuthorizedKeysCommand` OpenSSH.

### 11. Аутентификация LDAP / Active Directory

Аутентификация пользователей через LDAP-каталог или домен Active Directory вашей организации:

1. Перейдите в **Admin > Settings** и прокрутите до **LDAP / Active Directory Authentication**
2. Включите LDAP и заполните данные сервера:
   - **Server**: Имя хоста вашего LDAP-сервера (например, `dc01.corp.local`)
   - **Port**: 389 для LDAP, 636 для LDAPS
   - **SSL/TLS**: Включите для LDAPS или используйте StartTLS для обновления незащищённого соединения
3. Настройте сервисную учётную запись для поиска пользователей:
   - **Bind DN**: `CN=svc-git,OU=Service Accounts,DC=corp,DC=local`
   - **Bind Password**: Пароль сервисной учётной записи
4. Установите параметры поиска:
   - **Search Base DN**: `OU=Users,DC=corp,DC=local`
   - **User Filter**: `(sAMAccountName={0})` для AD, `(uid={0})` для OpenLDAP
5. Сопоставьте атрибуты LDAP с полями пользователя:
   - **Username**: `sAMAccountName` (AD) или `uid` (OpenLDAP)
   - **Email**: `mail`
   - **Display Name**: `displayName`
6. Опционально задайте **Admin Group DN** — участники этой группы автоматически повышаются до администратора
7. Нажмите **Test LDAP Connection** для проверки настроек
8. Сохраните настройки

Теперь пользователи могут входить с доменными учётными данными на странице входа. При первом входе автоматически создаётся локальная учётная запись с синхронизированными атрибутами из каталога. LDAP-аутентификация также используется для Git HTTP-операций (клонирование/отправка).

### 12. Секреты репозитория

Добавление зашифрованных секретов в репозитории для использования в CI/CD рабочих процессах:

1. Перейдите во вкладку **Settings** вашего репозитория
2. Прокрутите до карточки **Secrets** и нажмите **Add secret**
3. Введите имя (например, `DEPLOY_TOKEN`) и значение — значение шифруется с помощью AES-256
4. Секреты автоматически внедряются как переменные окружения в каждый запуск рабочего процесса

Ссылка на секреты в рабочем процессе:
```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy
        run: curl -H "Authorization: Bearer $DEPLOY_TOKEN" https://api.example.com/deploy
```

Поддерживаемые провайдеры: GitHub, Google, Microsoft, GitLab, Bitbucket, Facebook, Discord, Twitter/X.

Пользователи могут привязать несколько провайдеров к своей учётной записи в **Settings > Linked Accounts**.

### 16. CI/CD автоматический релиз

MyPersonalGit включает встроенный CI/CD конвейер, который автоматически тегирует, выпускает и отправляет Docker-образы при каждом push в main. Рабочие процессы автоматически запускаются при push — внешний CI-сервис не нужен.

**Как это работает:**
1. Push в `main` автоматически запускает `.github/workflows/release.yml`
2. Увеличивает патч-версию (`v1.15.1` -> `v1.15.2`), создаёт git-тег
3. Входит в Docker Hub, собирает образ и отправляет его как `:latest` и `:vX.Y.Z`

**Настройка:**
1. Перейдите в **Settings > Secrets** вашего репозитория в MyPersonalGit
2. Добавьте секрет с именем `DOCKERHUB_TOKEN` с вашим токеном доступа Docker Hub
3. Убедитесь, что контейнер MyPersonalGit имеет смонтированный Docker-сокет (`-v /var/run/docker.sock:/var/run/docker.sock`)
4. Отправьте в main — рабочий процесс запустится автоматически

**Совместимость с GitHub Actions:**
Тот же YAML рабочего процесса также работает в GitHub Actions — изменения не нужны. MyPersonalGit транслирует действия `uses:` в эквивалентные shell-команды во время выполнения:

| GitHub Action | Трансляция в MyPersonalGit |
|---|---|
| `actions/checkout@v4` | Репозиторий уже клонирован в `/workspace` |
| `actions/setup-dotnet@v4` | Установка .NET SDK через официальный скрипт |
| `actions/setup-node@v4` | Установка Node.js через NodeSource |
| `actions/setup-python@v5` | Установка Python через apt/apk |
| `actions/setup-java@v4` | Установка OpenJDK через apt/apk |
| `docker/login-action@v3` | `docker login` с паролем через stdin |
| `docker/build-push-action@v6` | `docker build && docker push` |
| `docker/setup-buildx-action@v3` | Не требуется (используется сборщик по умолчанию) |
| `softprops/action-gh-release@v2` | Создание реальной сущности Release в базе данных |
| `${{ secrets.X }}` | Переменная окружения `$X` |
| `${{ steps.X.outputs.Y }}` | Переменная окружения `$Y` |
| `${{ github.sha }}` | Переменная окружения `$GITHUB_SHA` |

**Параллельные задачи:**
Задачи выполняются параллельно по умолчанию. Используйте `needs:` для объявления зависимостей:
```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: dotnet build

  test:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: dotnet test

  deploy:
    needs: [build, test]
    runs-on: ubuntu-latest
    steps:
      - run: echo "deploying..."
```
Задачи без `needs:` запускаются немедленно. Задача отменяется, если любая из её зависимостей завершилась с ошибкой.

**Условные шаги:**
Используйте `if:` для управления выполнением шагов:
```yaml
steps:
  - name: Build
    run: dotnet build

  - name: Notify on failure
    if: failure()
    run: curl -X POST https://hooks.example.com/alert

  - name: Cleanup
    if: always()
    run: rm -rf ./tmp
```
Поддерживаемые выражения: `always()`, `success()` (по умолчанию), `failure()`, `cancelled()`, `true`, `false`.

**Выходные данные шагов:**
Шаги могут передавать значения последующим шагам через `$GITHUB_OUTPUT`:
```yaml
steps:
  - name: Determine version
    run: echo "version=1.2.3" >> $GITHUB_OUTPUT

  - name: Use version
    run: echo "Building version $version"
```

**Матричные сборки:**
Развёртывание задач по нескольким комбинациям с помощью `strategy.matrix`:
```yaml
jobs:
  test:
    strategy:
      fail-fast: true
      matrix:
        os: [ubuntu-latest, node-20]
        version: ["1.0", "2.0"]
    runs-on: ${{ matrix.os }}
    steps:
      - run: echo "Testing on ${{ matrix.os }} with version ${{ matrix.version }}"
```
Это создаёт 4 задачи: `test (ubuntu-latest, 1.0)`, `test (ubuntu-latest, 2.0)` и т.д. Все выполняются параллельно.

**Ручной запуск с параметрами (`workflow_dispatch`):**
Определение типизированных входных параметров, отображаемых как форма в UI при ручном запуске:
```yaml
on:
  workflow_dispatch:
    inputs:
      environment:
        description: "Target environment"
        required: true
        type: choice
        options:
          - staging
          - production
      debug:
        description: "Enable debug mode"
        type: boolean
        default: "false"

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying to $INPUT_ENVIRONMENT (debug=$INPUT_DEBUG)"
```
Значения параметров внедряются как переменные окружения `INPUT_<NAME>` (в верхнем регистре).

**Тайм-ауты задач:**
Установка `timeout-minutes` для задач для автоматического завершения при превышении лимита:
```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - run: make build
```
Тайм-аут по умолчанию — 360 минут (6 часов), как в GitHub Actions.

**Условия на уровне задач:**
Используйте `if:` на задачах для пропуска по условиям:
```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: dotnet test

  deploy:
    needs: test
    if: success()
    runs-on: ubuntu-latest
    steps:
      - run: echo "deploying..."

  notify:
    needs: test
    if: failure()
    runs-on: ubuntu-latest
    steps:
      - run: curl -X POST https://hooks.example.com/alert
```

**Выходные данные задач:**
Задачи могут передавать значения зависимым задачам через `outputs:`:
```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      version: ${{ steps.ver.outputs.version }}
    steps:
      - id: ver
        run: echo "version=1.2.3" >> $GITHUB_OUTPUT

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying version $version"
```

**Продолжение при ошибке:**
Разрешение сбоя шага без провала задачи:
```yaml
steps:
  - name: Optional lint
    continue-on-error: true
    run: npm run lint

  - name: Build (always runs)
    run: npm run build
```

**Фильтрация по путям:**
Запуск рабочих процессов только при изменении определённых файлов:
```yaml
on:
  push:
    branches: [main]
    paths:
      - 'src/**'
      - '*.csproj'
    # или используйте paths-ignore:
    # paths-ignore:
    #   - 'docs/**'
    #   - '*.md'
```

**Рабочая директория:**
Установка директории выполнения команд:
```yaml
defaults:
  run:
    working-directory: src/app

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: npm install          # выполняется в src/app
      - run: npm test
        working-directory: tests  # переопределяет значение по умолчанию
```

**Перезапуск рабочих процессов:**
Нажмите кнопку **Re-run** на любом завершённом, неудачном или отменённом запуске для создания нового запуска с теми же задачами, шагами и конфигурацией.

**Рабочие процессы для pull-запросов:**
Рабочие процессы с `on: pull_request` автоматически запускаются при создании не-черновика PR, выполняя проверки исходной ветки.

**Проверки статуса коммитов:**
Рабочие процессы автоматически устанавливают статусы коммитов (pending/success/failure), чтобы вы могли видеть результаты сборки в PR и применять обязательные проверки через защиту веток.

**Отмена рабочих процессов:**
Нажмите кнопку **Cancel** на любом выполняющемся или ожидающем рабочем процессе в интерфейсе Actions для немедленной остановки.

**Статусные значки:**
Встраивайте значки статуса сборки в README или куда угодно:
```markdown
![Build](http://your-server/api/badge/YourRepo/workflow)
![Status](http://your-server/api/badge/YourRepo/status)
```
Фильтрация по имени рабочего процесса: `/api/badge/YourRepo/workflow?workflow=Release%20%26%20Docker%20Push`

### 17. RSS/Atom ленты

Подписывайтесь на активность репозитория через стандартные Atom-ленты в любом RSS-ридере:

# Коммиты репозитория

## Настройка базы данных

MyPersonalGit по умолчанию использует **SQLite** — нулевая настройка, однофайловая база данных, идеально подходит для личного использования и небольших команд.

Для масштабных развёртываний (много одновременных пользователей, высокая доступность или если вы уже используете PostgreSQL) вы можете переключиться на **PostgreSQL**:

### Использование PostgreSQL

**Docker Compose** (рекомендуется для PostgreSQL):
```yaml
services:
  mypersonalgit:
    image: fennch/mypersonalgit:latest
    ports:
      - "8080:8080"
      - "2222:2222"
    environment:
      - Database__Provider=postgresql
      - ConnectionStrings__Default=Host=db;Database=mypersonalgit;Username=mypg;Password=secret
    depends_on:
      - db
    volumes:
      - repos:/repos

  db:
    image: postgres:17
    environment:
      - POSTGRES_DB=mypersonalgit
      - POSTGRES_USER=mypg
      - POSTGRES_PASSWORD=secret
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  repos:
  pgdata:
```

**Только переменные окружения** (если у вас уже есть PostgreSQL-сервер):
```bash
docker run -d --name mypersonalgit -p 8080:8080 \
  -v mypersonalgit-repos:/repos \
  -e Database__Provider=postgresql \
  -e ConnectionStrings__Default="Host=your-pg-server;Database=mypersonalgit;Username=mypg;Password=secret" \
  fennch/mypersonalgit:latest
```

Миграции EF Core выполняются автоматически при запуске для обоих провайдеров. Ручная настройка схемы не требуется.

### Переключение из панели администратора

Вы также можете переключить провайдер базы данных прямо из веб-интерфейса:

1. Перейдите в **Admin > Settings** — карточка **Database** находится вверху
2. Выберите **PostgreSQL** из выпадающего списка провайдеров
3. Введите строку подключения PostgreSQL (например, `Host=localhost;Database=mypersonalgit;Username=mypg;Password=secret`)
4. Нажмите **Save Database Settings**
5. Перезапустите приложение для применения изменений

Конфигурация сохраняется в `~/.mypersonalgit/database.json` (вне базы данных, чтобы её можно было прочитать до подключения).

### Выбор базы данных

| | SQLite | PostgreSQL |
|---|---|---|
| **Настройка** | Нулевая конфигурация (по умолчанию) | Требуется PostgreSQL-сервер |
| **Подходит для** | Личное использование, малые команды, NAS | Команды от 50+, высокая параллельность |
| **Резервное копирование** | Копирование файла `.db` | Стандартный `pg_dump` |
| **Параллелизм** | Один писатель (достаточно для большинства) | Полный мульти-доступ |
| **Миграция** | Н/Д | Смена провайдера + запуск приложения (автоматическая миграция) |

## Развёртывание на NAS

MyPersonalGit отлично работает на NAS (QNAP, Synology и т.д.) через Docker:

```bash
docker run -d --name mypersonalgit -p 8080:8080 -p 2222:2222 \
  -v /share/Container/mypersonalgit/repos:/repos \
  -v /share/Container/mypersonalgit/data:/data \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -e ConnectionStrings__Default="Data Source=/data/mypersonalgit.db" \
  -e Git__Users__admin=yourpassword \
  fennch/mypersonalgit:latest
```

Монтирование Docker-сокета опционально — нужно только если вы хотите запускать CI/CD рабочие процессы. Порт 2222 нужен только при включении встроенного SSH-сервера.

## Конфигурация

Все настройки можно конфигурировать в `appsettings.json`, через переменные окружения или через панель администратора по адресу `/admin`:

- Провайдер базы данных (SQLite или PostgreSQL)
- Корневая директория проектов
- Требования аутентификации
- Настройки регистрации пользователей
- Переключатели функций (Issues, Wiki, Projects, Actions)
- Максимальный размер репозитория и количество на пользователя
- Настройки SMTP для email-уведомлений
- Настройки push-уведомлений (Ntfy/Gotify)
- Встроенный SSH-сервер (включение/отключение, порт)
- Аутентификация LDAP/Active Directory (сервер, Bind DN, база поиска, фильтр пользователей, сопоставление атрибутов, группа администраторов)
- Конфигурация провайдеров OAuth/SSO (Client ID/Secret для каждого провайдера)
